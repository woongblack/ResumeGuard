## 0. 이 제품을 만드는 이유 (Why)

**직접적 동기:** 원티드 스타트업 백엔드/AI Product Engineer 포지션 지원 중, 서류 통과율 0%. 탈락 시 피드백이 없어 어느 부분이 문제인지 알 수 없음.

**만들고 쓴다:** 본인이 1번 사용자다. 포트폴리오 목적 + 실사용 병행. 서비스 출시 여부는 MVP 자기 검증 이후 결정.

## 1. 문제 정의 (Problem)

### 1-1. 직접 경험한 탈락 원인

| 문제 | 구체적 상황 |
|------|-----------|
| **수치적 성과 누락** | TPS, MAU, 개선율(%) 등 정량 지표 없음 |
| **프로젝트 심도 부족** | 기술 선택 이유, 트레이드오프 설명 없음 |
| **JD 언어 미스매치** | JD 키워드와 이력서 표현이 다름 |
| **AI 기술 스택 갭** | LangChain 등 AI/ML 도구 경험 부재 — AI PE 포지션 치명적 |

### 1-2. 구조적 문제

- 기업 AI 스크리닝 대규모 도입 중 (무하유 프리즘 기준 매년 120만 건 이상 자소서 분석 — 출처: [무하유 공식 자료](https://www.muhayu.com/?bmode=view&idx=15932071))
- 탈락 시 어떤 항목에서 감점됐는지 피드백 전혀 없음 — 채용 구조상 관행
- 결과: 같은 약점의 이력서를 반복 제출

### 1-3. 기존 해결책의 한계

| 서비스 | 한계 |
|--------|------|
| **Jobscan** | 영어 ATS 전용, 한국어 JD 대응 안 됨 |
| **무하유 프리즘** | B2B 기업용 — 지원자 직접 사용 불가 |
| **링커리어** | AI 분석 없음, 합격 자소서 참고 자료 플랫폼 |
| **ChatGPT 직접 사용** | 매번 프롬프트 작성, 포맷 불일정, URL 자동 파싱 불가 |

**결론:** 한국어 JD 기반 B2C AI 갭 분석 도구 없음.

## 3. 제품 정의 (What)

**ResumeGuard:** 채용공고 URL과 이력서 텍스트를 입력하면, 3개 에이전트가 JD와 이력서를 각각 분석하고 갭·압박 질문을 출력하는 웹 도구.

### ChatGPT 대비 차별화

| 항목 | ChatGPT | ResumeGuard |
|------|---------|-------------|
| JD 입력 | 직접 복붙 | URL → 자동 파싱 |
| 결과 포맷 | 매번 다름 | 3단계 판정 + 3항목 점수 + 갭 리스트 고정 |
| 수치 성과 체크 | 별도 지시 필요 | 기본 평가 항목 포함 |
| 면접 질문 | 일반적 | 이 서류의 빈틈을 겨냥한 압박 질문 |

### 한계 명시

실제 기업 AI 스크리닝 알고리즘은 공개되어 있지 않음. ResumeGuard는 **JD 적합도/기술 깊이/성과 수치** 기준으로 인사담당자 관점을 시뮬레이션하는 것임.

## 4. 시스템 아키텍처 (Architecture)

### 4-1. 에이전트 파이프라인

```
사용자 입력
 ├─ JD URL ──────→ [Agent 1: JD 분석]    ─┐
 │                  자격요건/우대사항/         ├→ [Agent 3: 갭 평가 + 피드백] → 최종 결과
 └─ 이력서 텍스트 → [Agent 2: 이력서 분석] ─┘
                    기술스택/경험/수치
```

- **Agent 1, 2**: 병렬 실행 (`asyncio.gather`) — 응답 시간 단축
- **Agent 3**: Agent 1, 2 결과를 입력으로 받아 갭 분석 + 점수 + 면접 질문 생성

### 4-2. JD URL 처리 방식

Playwright 크롤링 서버 없이 **Jina Reader API** 사용:

```
GET https://r.jina.ai/{jd_url}
→ 깔끔한 마크다운 텍스트 반환
→ Agent 1에 전달
```

- 별도 브라우저 자동화 불필요
- DOM 셀렉터 유지보수 없음
- 기본 무료

파싱 실패 시: 에러 코드 반환 + 사용자에게 수동 텍스트 입력 안내.

### 4-3. 데이터 흐름

```
[Frontend]
  → POST /analyze {jd_url, resume_text}
  → [FastAPI]
      → Jina Reader → JD 원문
      → asyncio.gather(Agent 1, Agent 2)
      → Agent 3(jd_structured, resume_structured)
  → JSON 응답
  → [Frontend] 시각화
```

## 5. 기능 요구사항 (Features)

### Phase 1 — MVP

| 우선순위 | 기능 | 요구사항 | 컴포넌트 |
|----------|------|---------|---------|
| Must | **JD URL 파싱** | Jina Reader로 URL 텍스트 추출 → Agent 1이 구조화. 실패 시 수동 입력 안내. | Backend |
| Must | **이력서 텍스트 입력** | 복사/붙여넣기 에디터 | Frontend (React) |
| Must | **3단계 판정 + 종합 점수** | 적합(PASS) / 검토필요(HOLD) / 부적합(FAIL) + 100점 만점 | Agent 3 |
| Must | **다면 평가 + 갭 추출** | 직무 적합도 / 기술 깊이 / 수치 성과 각 상/중/하. 이력서 텍스트 근거만 사용. | Agent 3 |
| Must | **면접 압박 질문 3개** | 이 서류의 구체적 빈틈을 겨냥. 일반 질문 금지. | Agent 3 |
| Must | **결과 시각화** | 3단계 배지 + 레이더 차트 + 강점/갭/질문 탭 | Frontend (React) |
| Must | **에러 처리** | Jina 실패 / LLM 타임아웃 / 빈 입력 각각 명시적 안내 | Backend / Frontend |

### Phase 2 (자기 검증 통과 후)

- PDF 업로드 (pypdf)
- 타 채용 플랫폼 지원 (사람인, 잡코리아 등)
- 평가 이력 저장 — **Phase 1 완료 전 DB 설계 금지**
- LangGraph 도입 — 조건부 분기(재시도 루프 등)가 필요해질 때

## 6. 평가 항목 상세 (Evaluation Spec)

### 3개 평가 축 (탈락 원인과 1:1 매핑)

| 항목 | 평가 기준 | 연결된 탈락 원인 |
|------|---------|----------------|
| **직무 적합도 (Job Fit)** | JD 자격요건/우대사항과 이력서 경험의 언어 매칭 | JD 언어 미스매치 |
| **기술 깊이 (Technical Depth)** | 기술 선택 이유, 트레이드오프, 문제 해결 과정 기술 여부 | 프로젝트 심도 부족 |
| **수치 성과 (Quantitative Evidence)** | TPS, MAU, 개선율 등 정량 지표 존재 여부 | 수치적 성과 누락 |

### 할루시네이션 방지 (시스템 프롬프트 강제)

> "서류에 텍스트로 명시되지 않은 기술, 경험, 수치는 존재하지 않는 것으로 간주한다. 추론이나 유추로 점수를 부여하지 않는다."

## 8. 기술 스택

| 레이어 | 기술 | 이유 |
|--------|------|------|
| Backend | Python + FastAPI | LLM SDK 연동 용이, 빠른 PoC |
| JD 파싱 | Jina Reader API | 크롤러 서버 불필요, 무료, 유지보수 없음 |
| LLM 추상화 | **LiteLLM** | 프로바이더 교체를 환경변수 1개로 처리 |
| 기본 모델 | **Gemini 3.1 Flash** | 무료 플랜 (1,500 req/day) — 개인 사용 충분 |
| 대체 모델 | Claude Sonnet / GPT-4o | 품질 비교 또는 필요 시 스왑 |
| 에이전트 실행 | `asyncio` | Agent 1·2 병렬화, 프레임워크 없이 단순 구현 |
| 에이전트 오케스트레이션 | LangGraph (Phase 2) | 조건부 분기 생길 때 도입 |
| Frontend | React + Recharts | 레이더 차트 라이브러리 성숙도 |
| DB | 없음 (Phase 1) | 복잡도 최소화 |

### 비용 구조

| 모델 | 비용 | 용도 |
|------|------|------|
| Gemini 3.1 Flash | 무료 (1,500 req/day) | 기본 — 분석 1회 = 호출 3번, 약 500회/일 가능 |
| Claude Sonnet | 유료 | 품질 테스트, 선택적 스왑 |

환경변수 하나로 모델 전환:
```bash
LLM_MODEL=gemini/gemini-3.1-flash   # 기본 (무료)
LLM_MODEL=claude-sonnet-4-6         # 교체 시
```

## 9. 성공 기준 (Phase 1 Exit Criteria)

| 지표 | 측정 방법 | 통과 기준 |
|------|----------|---------|
| JD 파싱 정확도 | 원티드 공고 20개 수동 검수 | 자격요건/우대사항/주요업무 분리 90% 이상 |
| 평가 일관성 | 동일 서류 3회 반복 | 점수 편차 ±5점 이내 |
| 자기 유용성 | 실제 지원 공고 5개에 직접 사용 | 결과가 실제 이력서 수정에 도움이 됐는가 (본인 판단) |

**실패 시 피벗:**
- 파싱 정확도 70% 미만 → Jina 대신 수동 JD 텍스트 입력으로 전환
- 자기 유용성 낮음 → 평가 항목 구조 재설계

## 10. API 스펙

**요청:**

```json
{
  "jd_url": "https://www.wanted.co.kr/wd/123456",
  "resume_text": "이력서 전문 텍스트..."
}
```

**정상 응답:**

```json
{
  "decision": "HOLD",
  "overall_score": 68,
  "evaluation_metrics": {
    "job_fit": "중",
    "technical_depth": "하",
    "quantitative_evidence": "하"
  },
  "summary": "Spring Boot 경험은 JD와 매칭되나, 기술 선택 근거와 수치 성과가 전무하여 기술 심도 검증 불가.",
  "strengths": [
    "JPA + MySQL 경험이 JD 자격요건과 직접 매칭됨"
  ],
  "gaps": [
    "TPS, MAU 등 수치 성과 지표 전혀 없음 — 기술 역량 판단 불가",
    "LangChain 등 AI 도구 경험 미기재 — AI PE 포지션 필수 항목",
    "기술 선택 이유와 트레이드오프 설명 없음"
  ],
  "expected_interview_questions": [
    "이력서에 수치 성과가 없는데, JPA 최적화 전후 성능을 측정한 적 있나요? 수치로 말해주세요.",
    "AI PE 포지션인데 LangChain 경험이 없습니다. AI 파이프라인을 직접 구축한 경험이 있나요?",
    "Spring Boot를 선택한 기술적 근거가 이력서에 없습니다. 다른 프레임워크 대신 선택한 이유가 뭔가요?"
  ]
}
```

**에러 응답:**

```json
{
  "error_code": "JD_FETCH_FAILED",
  "message": "채용공고를 불러올 수 없습니다. 공고가 마감되었거나 접근이 차단되었을 수 있습니다.",
  "suggestion": "JD 내용을 직접 텍스트로 붙여넣어 주세요."
}
```

| 에러 코드 | 원인 |
|----------|------|
| `JD_FETCH_FAILED` | Jina Reader 실패 (공고 만료, 차단) |
| `JD_PARSE_FAILED` | 텍스트 추출 성공 but 구조 미탐지 |
| `RESUME_EMPTY` | 이력서 입력값 없음 |
| `LLM_TIMEOUT` | LLM 응답 시간 초과 |
