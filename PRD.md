## 0. 이 제품을 만드는 이유 (Why)

**직접적 동기:** 원티드 스타트업 백엔드/AI Product Engineer 포지션 지원 중, 서류 통과율 0%. 탈락 시 피드백이 없어 어느 부분이 문제인지 알 수 없음.

**만들고 쓴다:** 본인이 1번 사용자다. 포트폴리오 목적 + 실사용 병행. 서비스 출시 여부는 MVP 자기 검증 이후 결정.

**방향 전환 기록 (v4→v5):** v4까지는 "JD-이력서 갭 분석기" 포지셔닝이었음. v5부터는 **기업 AI 스크리닝 재현**으로 목표를 상향. 이력서 단독 분석 → 이력서 + 자소서 복합 분석, 에이전트 3개 → 4개, 평가 축 3개 → 5개.

## 1. 문제 정의 (Problem)

### 1-1. 직접 경험한 탈락 원인

| 문제 | 구체적 상황 |
|------|-----------|
| **수치적 성과 누락** | TPS, MAU, 개선율(%) 등 정량 지표 없음 |
| **프로젝트 심도 부족** | 기술 선택 이유, 트레이드오프 설명 없음 |
| **JD 언어 미스매치** | JD 키워드와 이력서 표현이 다름 |
| **AI 기술 스택 갭** | LangChain 등 AI/ML 도구 경험 부재 — AI PE 포지션 치명적 |
| **자소서 BP 부재** | 경험 나열 수준. 상황/행동/결과 구조 없음 |

### 1-2. 구조적 문제

- 기업 AI 스크리닝 대규모 도입 중 (무하유 프리즘 기준 매년 120만 건 이상 자소서 분석 — 출처: [무하유 공식 자료](https://www.muhayu.com/?bmode=view&idx=15932071))
- 무하유 프리즘이 실제로 하는 것: 자소서 역량(BP) 평가, AI 생성 탐지, JD 적합도, 3단계 분류
- 탈락 시 어떤 항목에서 감점됐는지 피드백 전혀 없음 — 채용 구조상 관행
- 결과: 같은 약점의 서류를 반복 제출

### 1-3. 기존 해결책의 한계

| 서비스 | 한계 |
|--------|------|
| **Jobscan** | 영어 ATS 전용, 한국어 JD 대응 안 됨 |
| **무하유 프리즘** | B2B 기업용 — 지원자 직접 사용 불가, 알고리즘 비공개 |
| **링커리어** | AI 분석 없음, 합격 자소서 참고 자료 플랫폼 |
| **ChatGPT 직접 사용** | 매번 프롬프트 작성, 포맷 불일정, URL 자동 파싱 불가 |

**결론:** 한국어 JD 기반 B2C AI 스크리닝 도구 없음.

## 3. 제품 정의 (What)

**ResumeGuard:** 채용공고 URL + 이력서 + 자소서를 입력하면, 4개 에이전트가 JD·이력서·자소서를 분석하고 기업 AI 스크리닝 방식으로 종합 판정과 피드백을 출력하는 웹 도구.

### 포지셔닝

기업 AI 스크리닝(무하유 프리즘 등)의 **공개된 평가 기준**을 기반으로 동일한 평가를 지원자 관점에서 먼저 수행. 알고리즘 자체는 비공개이므로 완전한 재현이 아닌 **동등한 평가 기준 적용**이 목표.

### ChatGPT 대비 차별화

| 항목 | ChatGPT | ResumeGuard |
|------|---------|-------------|
| JD 입력 | 직접 복붙 | URL → 자동 파싱 |
| 평가 대상 | 이력서 또는 자소서 (분리) | 이력서 + 자소서 통합 평가 |
| 결과 포맷 | 매번 다름 | 5개 축 점수 + 3단계 판정 + 갭 + 질문 고정 |
| AI 생성 탐지 | 없음 | 포함 |

### 한계 명시

- 무하유 프리즘의 실제 알고리즘과 가중치는 비공개 → 완전한 재현 불가
- 표절 검사는 타 제출 서류 DB가 없으면 기술적으로 불가 → **구현 제외**
- AI 생성 탐지는 확률 기반 — 오탐/미탐 존재, 참고 수준으로 제공

## 4. 시스템 아키텍처 (Architecture)

### 4-1. 에이전트 파이프라인

```
사용자 입력
 ├─ JD URL ──────→ [Agent 1: JD 분석]         ─┐
 ├─ 이력서 텍스트 → [Agent 2: 이력서 분석]      ├→ [Agent 4: 종합 판정 + 피드백] → 최종 결과
 └─ 자소서 텍스트 → [Agent 3: 자소서 BP 평가]  ─┘
                    + AI 생성 탐지 (병렬)
```

- **Agent 1, 2, 3**: 병렬 실행 (`asyncio.gather`) — 응답 시간 단축
- **Agent 4**: Agent 1·2·3 + AI 탐지 결과를 입력으로 받아 종합 판정 + 5개 축 점수 + 면접 질문 생성

### 4-2. 에이전트 역할 상세

| 에이전트 | 입력 | 출력 |
|----------|------|------|
| **Agent 1: JD 분석** | JD URL → Jina Reader 변환 텍스트 | 자격요건 / 우대사항 / 주요업무 구조화 |
| **Agent 2: 이력서 분석** | 이력서 텍스트 | 기술스택 / 프로젝트 경험 / 수치 지표 구조화 |
| **Agent 3: 자소서 BP 평가** | 자소서 텍스트 | STAR 구조 분석 + 핵심 역량(문제해결/협업/주인의식/학습) 평가 |
| **AI 탐지 모듈** | 자소서 텍스트 | AI 생성 의심도 (0–100) |
| **Agent 4: 종합 판정** | Agent 1·2·3 결과 + AI 탐지 점수 | 3단계 판정 + 5개 축 점수 + 갭 + 면접 질문 |

### 4-3. JD URL 처리 방식

**지원 플랫폼 (Phase 1):** 원티드, 사람인, 점핏

Playwright 크롤링 서버 없이 **Jina Reader API** 사용:

```
GET https://r.jina.ai/{jd_url}
→ 깔끔한 마크다운 텍스트 반환
→ Agent 1에 전달
```

| 플랫폼 | URL 패턴 예시 | Jina Reader 실측 결과 |
|--------|-------------|----------------------|
| 원티드 | `https://www.wanted.co.kr/wd/{id}` | ✅ 동작 확인. 주요업무/자격요건/우대사항 추출됨. 네비게이션·이미지 태그 노이즈 포함 → Agent 1 프롬프트에서 필터링 |
| 사람인 | `https://www.saramin.co.kr/zf_user/jobs/relay/view?rec_idx={id}` | ✅ 동작 확인. 자격요건/우대사항 추출됨. 로그인 유도·통계 UI 노이즈 포함 → Agent 1 프롬프트에서 필터링 |
| 점핏 | `https://jumpit.saramin.co.kr/position/{id}` | ✅ 동작 확인. 주요업무/자격요건/우대사항/복지 추출됨. 기술스택 이미지 태그 노이즈 포함 → Agent 1 프롬프트에서 필터링 |

**공통 fallback:** Jina Reader 실패 시 `JD_FETCH_FAILED` 에러 코드 반환 + 수동 텍스트 입력 안내.

### 4-4. AI 생성 탐지 방식

- **1차**: GPTZero API (외부 전문 탐지 서비스)
- **Fallback**: LLM 프롬프트 기반 탐지 (GPTZero 호출 실패 시)
- **출력**: 의심도 점수(0–100) + 판정 (낮음/보통/높음) + 참고 경고 문구

GPTZero API 키 없을 경우 LLM fallback으로 자동 전환.

### 4-5. 데이터 흐름

```
[Frontend]
  → POST /analyze {jd_url, resume_text, cover_letter_text}
  → [FastAPI]
      → Jina Reader → JD 원문
      → asyncio.gather(Agent 1, Agent 2, Agent 3, AI 탐지)
      → Agent 4(jd_structured, resume_structured, cover_letter_evaluated, ai_score)
  → JSON 응답
  → [Frontend] 시각화
```

## 5. 기능 요구사항 (Features)

### Phase 1 — MVP

| 우선순위 | 기능 | 요구사항 | 컴포넌트 |
|----------|------|---------|---------|
| Must | **JD URL 파싱** | Jina Reader로 URL 텍스트 추출 → Agent 1 구조화. 실패 시 수동 입력 안내. | Backend |
| Must | **이력서 입력** | PDF 업로드 (pypdf 텍스트 추출) 또는 텍스트 직접 붙여넣기. 추출된 텍스트는 브라우저 localStorage에 저장 → 다음 접속 시 자동 불러오기. | Frontend + Backend |
| Must | **자소서 입력** | PDF 업로드 또는 텍스트 붙여넣기 (선택 입력). 동일하게 localStorage 저장. 없으면 자소서 관련 축 평가 생략. | Frontend + Backend |
| Must | **3단계 판정 + 종합 점수** | 적합(PASS) / 검토필요(HOLD) / 부적합(FAIL) + 100점 만점 | Agent 4 |
| Must | **5개 축 평가 + 갭 추출** | 직무 적합도 / 기술 깊이 / 수치 성과 / 자소서 역량 / AI 생성 의심도. 이력서·자소서 텍스트 근거만 사용. | Agent 3·4 |
| Must | **자소서 BP 평가** | STAR 구조 존재 여부 + 핵심 역량(문제해결/협업/주인의식/학습) 상/중/하 | Agent 3 |
| Must | **AI 생성 탐지** | 의심도 점수 + 판정 + 참고 경고. 오탐 가능성 명시. | AI 탐지 모듈 |
| Must | **면접 압박 질문 3개** | 이 서류의 구체적 빈틈을 겨냥. 이력서·자소서 통합 근거. | Agent 4 |
| Must | **결과 시각화** | 3단계 배지 + 레이더 차트(5개 축) + 강점/갭/질문/AI탐지 탭 | Frontend |
| Must | **에러 처리** | Jina 실패 / LLM 타임아웃 / 빈 입력 각각 명시적 안내 | Backend / Frontend |

### Phase 2 (자기 검증 통과 후)

- 추가 플랫폼 지원 (잡코리아 등)
- 평가 이력 저장 — **Phase 1 완료 전 DB 설계 금지**
- LangGraph 도입 — 조건부 분기(재시도 루프 등)가 필요해질 때

## 6. 평가 항목 상세 (Evaluation Spec)

### 5개 평가 축

| 항목 | 평가 기준 | 근거 서류 | 연결된 문제 |
|------|---------|----------|------------|
| **직무 적합도 (Job Fit)** | JD 자격요건/우대사항과 이력서 경험의 언어 매칭 | 이력서 | JD 언어 미스매치 |
| **기술 깊이 (Technical Depth)** | 기술 선택 이유, 트레이드오프, 문제 해결 과정 기술 여부 | 이력서 | 프로젝트 심도 부족 |
| **수치 성과 (Quantitative Evidence)** | TPS, MAU, 개선율 등 정량 지표 존재 여부 | 이력서 | 수치 성과 누락 |
| **자소서 역량 (Cover Letter BP)** | STAR 구조 + 문제해결/협업/주인의식/학습 역량 기술 여부 | 자소서 | 자소서 BP 부재 |
| **AI 생성 의심도 (AI Detection)** | GPTZero API 기반 점수 (0–100) | 자소서 | 기업 AI 탐지 대비 |

### 자소서 역량 세부 기준 (STAR)

| 역량 | 평가 기준 |
|------|---------|
| **문제 해결력** | 구체적 문제 상황(S) + 본인 행동(A) + 정량 결과(R) 기술 여부 |
| **협업/의사소통** | 팀 내 역할, 갈등 해결 경험 기술 여부 |
| **주인의식** | 자발적 개선 시도 또는 책임 완수 경험 기술 여부 |
| **학습 능력** | 새 기술/환경 습득 과정과 적용 결과 기술 여부 |

### 할루시네이션 방지 (시스템 프롬프트 강제)

> "서류에 텍스트로 명시되지 않은 기술, 경험, 수치, 역량은 존재하지 않는 것으로 간주한다. 추론이나 유추로 점수를 부여하지 않는다."

## 8. 기술 스택

| 레이어 | 기술 | 이유 |
|--------|------|------|
| Backend | Python + FastAPI | LLM SDK 연동 용이, 빠른 PoC |
| JD 파싱 | Jina Reader API | 크롤러 서버 불필요, 무료, 유지보수 없음 |
| LLM 추상화 | **LiteLLM** | 프로바이더 교체를 환경변수 1개로 처리 |
| 기본 모델 | **Gemini 2.5 Flash** | 무료 플랜 (1,500 req/day) — 분석 1회 = 호출 4번 → 약 375회/일 |
| 대체 모델 | Claude Sonnet / GPT-4o | 품질 비교 또는 필요 시 스왑 |
| 에이전트 실행 | `asyncio` | Agent 1·2·3 병렬화, 프레임워크 없이 단순 구현 |
| 에이전트 오케스트레이션 | LangGraph (Phase 2) | 조건부 분기 생길 때 도입 |
| AI 생성 탐지 | GPTZero API + LLM fallback | 전문 탐지 서비스, fallback으로 비용 제어 |
| Frontend | React + Recharts | 레이더 차트 라이브러리 성숙도 |
| DB | 없음 (Phase 1) | 복잡도 최소화 |

### 비용 구조

| 항목 | 비용 | 한도 |
|------|------|------|
| Gemini 2.5 Flash | 무료 | 1,500 req/day |
| GPTZero API | 무료 플랜 | 월 10,000 단어 |
| Jina Reader API | 무료 | - |

환경변수 하나로 모델 전환:
```bash
LLM_MODEL=gemini/gemini-3.1-flash   # 기본 (무료)
LLM_MODEL=claude-sonnet-4-6         # 교체 시
```

## 9. 성공 기준 (Phase 1 Exit Criteria)

| 지표 | 측정 방법 | 통과 기준 |
|------|----------|---------|
| JD 파싱 정확도 | 원티드·사람인·점핏 공고 각 7개 이상 (총 20개 이상) 수동 검수 | 자격요건/우대사항/주요업무 분리 90% 이상. Jina 실패 시 수동 입력 fallback 포함 측정. |
| 이력서 평가 일관성 | 동일 서류 3회 반복 | 점수 편차 ±5점 이내 |
| 자소서 BP 평가 일관성 | 동일 자소서 3회 반복 | 역량 판정(상/중/하) 2회 이상 일치 |
| AI 탐지 참고성 | 본인이 작성한 자소서 vs. ChatGPT 생성 자소서 | 방향 구분(낮음 vs. 높음) 일치 여부 |
| 자기 유용성 | 실제 지원 공고 5개에 직접 사용 | 결과가 실제 이력서/자소서 수정에 도움이 됐는가 (본인 판단) |

**실패 시 피벗:**
- 파싱 정확도 70% 미만 → Jina 대신 수동 JD 텍스트 입력으로 전환
- 자소서 BP 평가 일관성 저조 → Agent 3 프롬프트 재설계 또는 평가 축 단순화
- AI 탐지 신뢰도 낮음 → 탐지 탭 제거, "참고 기능" 레이블로 강등
