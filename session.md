## 아키텍처 결정 사항 (확정)

### 1. Playwright 크롤링 서버 → Jina Reader API

**이유:** URL을 받아도 결국 크롤링이 필요하지만, Playwright 서버를 직접 운영하면:
- DOM 셀렉터 유지보수 부담
- 원티드 구조 변경 시 즉시 깨짐
- 법적 리스크 (이용약관)

**대안:** Jina Reader API — `GET https://r.jina.ai/{url}` 한 줄로 어떤 URL이든 마크다운 변환. 무료. 유지보수 없음.

### 2. LangChain 제거 → LiteLLM

**이유:** 멀티 프로바이더 추상화가 목적이라면 LangChain 전체가 불필요. LiteLLM이 이 목적에 특화되어 있고 훨씬 가벼움.

```python
from litellm import completion
response = completion(model="gemini/gemini-3.1-flash", messages=[...])
# 환경변수 하나로 모델 교체 가능
```

### 3. 기본 모델: Gemini 3.1 Flash (무료)

**이유:** Claude는 무조건 유료. Gemini 3.1 Flash는 무료 플랜 1,500 req/day — 분석 1회 = LLM 호출 3번 → 약 500회/일 가능. 개인 사용 충분.

```bash
LLM_MODEL=gemini/gemini-3.1-flash   # 기본 (무료)
LLM_MODEL=claude-sonnet-4-6         # 교체 시
```

### 4. 3-에이전트 파이프라인

```
사용자 입력
 ├─ JD URL ──────→ [Agent 1: JD 분석]     ─┐
 │                  자격요건/우대사항 구조화    ├→ [Agent 3: 갭 평가 + 피드백]
 └─ 이력서 텍스트 → [Agent 2: 이력서 분석] ─┘
                    기술스택/경험/수치 구조화
```

- Agent 1, 2: **asyncio.gather로 병렬 실행** (응답 시간 단축)
- Agent 3: 1, 2 결과를 받아 갭 분석 + 점수 + 면접 질문 생성

### 5. LangGraph: Phase 2로 연기

**이유:** 현재 파이프라인은 DAG 구조 (분기 없음, 루프 없음). asyncio로 충분. LangGraph는 조건부 분기(실패 시 재시도 루프 등)가 생길 때 도입.

### 6. DB: Phase 1 없음

평가 결과는 세션 메모리에만 보관. MySQL + SQLAlchemy는 Phase 1 검증 통과 후 도입. **Phase 1 완료 전 DB 설계 시작 금지.**

## 마지막 세션에서 제기된 미결 질문

### ❗ 핵심 방향 미결정

세션 마지막에 중요한 질문이 제기됨:

> "진짜 실제 기업 AI 검수 기능을 똑같이 재현하고 싶은데, 이에 맞게 PRD 작성하고 있는건가?"

**현재 PRD의 솔직한 평가:**
- 현재 PRD는 **JD-이력서 갭 분석기**에 가까움
- 기업 AI 스크리닝 (무하유 프리즘)의 실제 기능과 차이가 있음

| 기업 AI가 하는 것 | 현재 PRD |
|-----------------|---------|
| 자소서 역량 평가 (BP 평가) | ❌ 이력서만 |
| AI 생성 탐지 | ❌ 없음 |
| 표절 검사 | ❌ 불가 (데이터 없음) |
| 3단계 분류 | ✅ |
| JD 적합도 분석 | ✅ |

**선택지 두 가지 (아직 결정 안 됨):**

**A. 현재 방향 유지 — JD-이력서 갭 분석기**
- 기업 AI와 비슷한 출력 포맷 (3단계 판정)
- 실제 취업에 유용
- 재현이 아니라 시뮬레이션으로 포지셔닝

**B. 기업 AI 재현 시도 — 스코프 확대**
- 이력서 + 자소서 둘 다 분석
- AI 생성 탐지 추가
- HR 역량 프레임워크 도입
- 훨씬 복잡, MVP로 완성 어려움

**다음 세션 시작 시 이 질문부터 결정해야 함.**

## Mirror Agent 철학 적용 기록

이 PRD는 [mirror-agent](https://github.com/woongblack/mirror-agent.git)의 8개 규칙을 적용하여 작성됨.

적용된 주요 규칙:
- **Rule 2 (차별화 명시):** ChatGPT 대비 차별화 표 추가, 기업 AI와 "동일한 메커니즘" 표현 → "시뮬레이션"으로 수정
- **Rule 4 (사용자 니즈 증거):** 본인 직접 경험을 1차 근거로 명시
- **Rule 5 (1인 운영 가능성):** Playwright 제거, DB Phase 1 금지
- **Rule 8 (동기의 정직성):** 섹션 0에 포트폴리오 + 실사용 목적 명시

## Phase 1 성공 기준

| 지표 | 통과 기준 |
|------|---------|
| JD 파싱 정확도 | 원티드 공고 20개 수동 검수, 90% 이상 |
| 평가 일관성 | 동일 서류 3회 반복, 점수 편차 ±5점 이내 |

**실패 시 피벗:**
- 파싱 정확도 70% 미만 → Jina 대신 수동 텍스트 입력 전환
- 유용성 낮음 → 평가 항목 구조 재설계
