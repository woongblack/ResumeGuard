## 아키텍처 결정 사항 (확정)

### 1. JD 파싱: Jina Reader API

`GET https://r.jina.ai/{url}` 한 줄로 어떤 URL이든 마크다운 변환. 무료. 유지보수 없음.

**실측 결과 (2026-05-26):**

| 플랫폼 | 결과 | 노이즈 |
|--------|------|--------|
| 원티드 | ✅ 동작 | 네비게이션 메뉴, 이미지 태그 |
| 사람인 | ✅ 동작 | 로그인 유도, 합격률 통계 UI |
| 점핏 | ✅ 동작 | 기술스택 이미지 태그 |

노이즈는 Agent 1 프롬프트에서 "자격요건/우대사항/주요업무만 추출" 지시로 처리.

### 2. LLM 추상화: LiteLLM

```python
from litellm import acompletion
response = await acompletion(model="gemini/gemini-2.5-flash", messages=[...])
# 환경변수 하나로 모델 교체 가능
```

### 3. 기본 모델: Gemini 2.5 Flash (무료)

무료 플랜 1,500 req/day. 분석 1회 = LLM 호출 4번 → 약 375회/일 가능.

```bash
LLM_MODEL=gemini/gemini-2.5-flash   # 기본 (무료)
LLM_MODEL=claude-sonnet-4-6         # 교체 시
```

### 4. 4-에이전트 파이프라인

```
사용자 입력
 ├─ JD URL ──────→ [Agent 1: JD 분석]         ─┐
 ├─ 이력서 PDF/텍스트 → [Agent 2: 이력서 분석]  ├→ [Agent 4: 종합 판정]
 └─ 자소서 PDF/텍스트 → [Agent 3: 자소서 BP]  ─┘
                         + AI 생성 탐지 (병렬)
```

- Agent 1, 2, 3 + AI 탐지: **asyncio.gather로 병렬 실행**
- Agent 4: 1·2·3 + AI 탐지 결과 → 종합 판정 + 5개 축 점수 + 면접 질문

### 5. 입력 저장: 브라우저 localStorage

PDF 업로드 → pypdf 텍스트 추출 → localStorage 저장 → 다음 접속 시 자동 불러오기.
서버 DB, 로그인 불필요. 혼자 쓰는 MVP에 충분.

### 6. DB: Phase 1 없음

평가 결과는 세션 메모리에만 보관. MySQL + SQLAlchemy는 Phase 1 검증 통과 후 도입.

### 7. LangGraph: Phase 2로 연기

현재 파이프라인은 DAG 구조. asyncio로 충분. 조건부 분기 생길 때 도입.

## 평가 항목 5개 축 (탈락 원인과 매핑)

| 항목 | 평가 기준 | 근거 서류 |
|------|---------|----------|
| 직무 적합도 (Job Fit) | JD 자격요건/우대사항과 이력서 언어 매칭 | 이력서 |
| 기술 깊이 (Technical Depth) | 기술 선택 이유, 트레이드오프 기술 여부 | 이력서 |
| 수치 성과 (Quantitative Evidence) | TPS, MAU, 개선율 등 정량 지표 존재 여부 | 이력서 |
| 자소서 역량 (Cover Letter BP) | STAR 구조 + 문제해결/협업/주인의식/학습 | 자소서 |
| AI 생성 의심도 (AI Detection) | GPTZero API 점수 (0–100) | 자소서 |

**할루시네이션 방지:** "서류에 텍스트로 명시되지 않은 기술/경험/수치/역량은 존재하지 않는 것으로 간주. 추론으로 점수 부여 금지."

## 사용자 흐름

```
[1. 입력]
  ├─ JD URL 붙여넣기 (원티드/사람인/점핏)
  ├─ 이력서 PDF 업로드 또는 텍스트 붙여넣기 (localStorage 자동 저장)
  └─ 자소서 PDF 업로드 또는 텍스트 붙여넣기 (선택, localStorage 자동 저장)

         ↓ 분석 버튼 클릭

[2. 백엔드 처리 — 병렬]
  Jina Reader → JD 텍스트 추출
  ├─ Agent 1: JD 구조화
  ├─ Agent 2: 이력서 구조화
  ├─ Agent 3: 자소서 BP 평가
  └─ AI 탐지: 자소서 AI 생성 의심도

         ↓ 병렬 완료 후

  Agent 4: 종합 판정

[3. 결과 화면]
  ├─ 3단계 배지: PASS / HOLD / FAIL + 종합 점수
  ├─ 레이더 차트: 5개 축 시각화
  └─ 탭: 강점 / 갭 / 면접 압박 질문 3개 / AI 탐지 결과
```

## Phase 1 성공 기준

| 지표 | 측정 방법 | 통과 기준 |
|------|----------|---------|
| JD 파싱 정확도 | 원티드·사람인·점핏 각 7개 이상 (총 20개) 수동 검수 | 자격요건/우대사항/주요업무 분리 90% 이상 |
| 이력서 평가 일관성 | 동일 서류 3회 반복 | 점수 편차 ±5점 이내 |
| 자소서 BP 평가 일관성 | 동일 자소서 3회 반복 | 역량 판정(상/중/하) 2회 이상 일치 |
| AI 탐지 참고성 | 본인 작성 vs. ChatGPT 생성 자소서 | 방향 구분(낮음 vs. 높음) 일치 여부 |
| 자기 유용성 | 실제 지원 공고 5개에 직접 사용 | 결과가 이력서/자소서 수정에 도움이 됐는가 (본인 판단) |

**실패 시 피벗:**
- 파싱 정확도 70% 미만 → Jina 대신 수동 JD 텍스트 입력 전환
- 자소서 BP 일관성 저조 → Agent 3 프롬프트 재설계
- AI 탐지 신뢰도 낮음 → 탐지 탭 "참고 기능"으로 강등

## Mirror Agent 철학 적용 기록

이 PRD는 [mirror-agent](https://github.com/woongblack/mirror-agent.git)의 8개 규칙을 적용하여 작성됨.

- **Rule 2 (차별화 명시):** ChatGPT 대비 차별화 표 추가, 기업 AI와 "동일한 메커니즘" → "동등한 평가 기준 적용"으로 수정
- **Rule 4 (사용자 니즈 증거):** 본인 직접 경험을 1차 근거로 명시
- **Rule 5 (1인 운영 가능성):** Playwright 제거, DB Phase 1 금지, localStorage 활용
- **Rule 8 (동기의 정직성):** 섹션 0에 포트폴리오 + 실사용 목적 명시
